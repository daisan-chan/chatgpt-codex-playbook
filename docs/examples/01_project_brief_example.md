# 01 プロジェクト概要サンプル

この文書は `docs/02_project_brief_template.md` の記入例です。書き方と粒度の参考を目的としており、実際のプロジェクトでは項目の増減や詳細度の調整を行って構いません。

## プロジェクト名
- 運用ドキュメントのレビューフロー整理ミニ改訂

## 目的
- ドキュメント変更時のレビュー手順を、誰が見ても同じ流れで実行できる状態にする。
- レビュー観点の抜け漏れを減らし、差分確認の往復回数を減らす。

## 背景
- 現状はレビュー観点が担当者ごとに揺れ、同じ指摘が繰り返されることがある。
- 変更の粒度は小さいが、確認手順が明文化されていないため判断に時間がかかる。

## 対象範囲
- `docs/04_review_checklist.md` の観点整理
- PR 説明で最低限記載する確認項目の明確化
- main thread へ戻す報告内容の統一

## 対象外
- 実装コードや CI 設定の変更
- 新しい運用ツールの導入
- テンプレート体系の全面再設計

## 関係者
- 人: 目的・優先順位・最終承認を担当し、採用可否を決定する。
- ChatGPT: 論点整理、文案作成、レビュー観点の抜け漏れ確認を支援する。
- Codex: 指示に基づくファイル編集、差分提示、作業結果の報告を担当する。

## 成果物
- 更新済みレビュー関連ドキュメント 1〜2 ファイル
- 変更理由と確認結果を含む PR 説明
- main thread へ戻す要約報告（採用内容・未解決事項・次ステップ）

## 制約
- 変更はドキュメントのみとする。
- 1 回の変更対象はレビュー可能な小さな範囲に限定する。
- 既存の命名規則と docs の連番構成を崩さない。

## 完了条件
- レビュー観点が重複なく整理され、チェック時の迷いが減る。
- PR 説明に目的・差分・確認結果が記載されている。
- main thread へ戻す報告に、採用内容・未解決事項・次ステップが含まれている。

## ChatGPT / Codex 引き継ぎメモ
- ChatGPT に期待すること:
  - 現行ドキュメントの論点整理と、最小変更での改訂案提示。
  - 修正案が対象範囲外へ広がっていないかの確認。
- Codex に渡す実行指示:
  - 対象ファイルを明示し、見出し単位で修正範囲を指定する。
  - 変更後は差分要約と実施確認コマンドを簡潔に報告する。
- レビュー時に重視する観点:
  - 目的に対して変更が過不足ないか。
  - 用語と役割分担（人 / ChatGPT / Codex）の整合が保たれているか。
  - 次の小さな作業へ自然に接続できる報告になっているか。

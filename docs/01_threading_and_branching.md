# 01 スレッド運用とブランチ運用

この文書は、上位幹 / 中間幹 / 枝の役割差を整理し、Git ブランチ・プルリクとの対応をそろえるための運用ガイドです。何をどの場で扱うかを明確にすることで、判断の混線を防ぎ、Codex に渡す指示を小さく具体化しやすくします。

## 上位幹とは
上位幹は、プロジェクト全体の判断を集約する正本です。目的・優先順位・完了条件・横断制約をここにそろえることで、下位スレッドごとの前提ぶれを防ぎます。

上位幹では、全体に関わる判断を扱います。詳細作業や試行錯誤のログは抱え込まず、必要に応じて中間幹や枝へ切り出します。

## 中間幹とは
中間幹は、1 テーマを継続管理するための会話スレッドです。上位幹から見ればテーマ別の整理場所であり、枝から見れば戻し先になります。

中間幹は、必ず作るものではありません。**1 テーマを継続管理する必要が生じたら**作ります。

## 枝とは
枝は、1 つの具体タスクを進めるための作業スレッドです。上位幹や中間幹で決まった前提を受けて、対象範囲・変更内容・確認方法を具体化します。

枝では、具体化・詳細化・試行錯誤・Codex 実行依頼を扱います。複数テーマの継続管理や全体優先順位の再設計は扱いません。

## 3 層の使い分け
### 上位幹で扱うこと
- プロジェクト全体の方針
- テーマ間の優先順位
- 完了条件や制約の更新
- 中間幹を作るかどうかの判断

### 中間幹で扱うこと
- 1 テーマの継続管理
- 関連する枝の整理
- テーマ内の保留事項、未解決事項、次の枝候補
- 上位幹へ戻すべき論点の要約

### 枝で扱うこと
- 1 つの具体タスク
- 文書差分の作成や比較案の検討
- Codex への実行指示の具体化
- 作業結果の要約と親への戻し

## 中間幹新設トリガー
次のいずれかが見えたら、中間幹の新設を検討します。

- 同一テーマで枝が 2 本以上になりそう
- 上位幹にテーマ別の進捗管理が流れ込み始めた
- すぐには結論が出ない保留論点を継続管理したい
- テーマ単位で戻し先を固定したほうが読みやすい

中間幹を作る目的は、情報を増やすことではなく、**判断の種類を分けること**です。上位幹を軽く保ちつつ、テーマの継続管理を安定させるために使います。

## 枝化トリガー
次のいずれかが見えたら、枝への切り出しを優先します。

- 具体的な修正文案や作業手順を詰め始めた
- 代替案比較や試行錯誤が必要になった
- Codex に渡す対象ファイルや完了条件を具体化し始めた
- 親スレッドで詳細ログを追うのが読みにくくなってきた

## 幹 / 中間幹 / 枝 の対比ルール
| 観点 | 上位幹 | 中間幹 | 枝 |
| --- | --- | --- | --- |
| 主な役割 | 全体方針と優先順位 | 1 テーマの継続管理 | 1 具体タスクの実行 |
| 期間 | プロジェクト全体 | テーマ継続中 | タスク完了まで |
| 扱う判断 | 横断判断 | テーマ判断 | 実行判断 |
| 主な戻し先 | 人の最終判断 | 上位幹 | 中間幹または上位幹 |
| 抱えないもの | 詳細作業 | 全体優先順位 | 全体方針、複数テーマ管理 |

## 自己認識の最小セット
新しいスレッドを始めるときは、少なくとも次を明示します。

- 自身の階層
- 親スレッド
- テーマ
- 目的
- 扱う範囲
- 扱わない範囲
- 戻し先
- 終着点

テンプレートは `docs/06_thread_operation_templates.md` を参照してください。

## ChatGPT が行う移行判断
ChatGPT は、会話内容が階層に合っていないとき、次のような移行提案を行います。

- 上位幹に詳細作業が入ったら、枝化または中間幹化を提案する
- 中間幹で 1 具体タスクの深掘りが始まったら、枝化を提案する
- 枝で全体方針の再設計が必要になったら、親へ戻すことを提案する
- スレッド名が実態からずれたら、命名変更を提案する

## ChatGPT スレッドと Git ブランチの関係
この節では、会話上の作業単位と Git 上の変更単位をどう対応づけるかを整理します。1 つの枝スレッドに 1 つの Git ブランチを対応させると、何の議論からどの差分が生まれたかを追跡しやすくなります。プルリクは提出の手続きではなく、枝スレッドの成果をレビュー可能な形にする受け皿として扱います。

- 1 つの枝スレッドに対して 1 つの Git ブランチを基本とします
- スレッドで決めた変更範囲を、同名または対応が分かるブランチ名に反映します
- 中間幹や上位幹は会話の整理軸であり、通常は Git ブランチそのものには対応させません
- プルリクは枝スレッドの成果物として扱います

## 戻し方の最小セット
### 枝 → 中間幹
- 何を試したか
- 何を採用したか
- 未解決事項
- 次に必要な枝または判断

### 中間幹 → 上位幹
- テーマ全体で何が前進したか
- 今回判断してほしいこと
- 優先順位や制約に影響すること
- 次の候補タスク

### 枝 → 上位幹
中間幹がない場合は、枝から上位幹へ直接戻します。この場合も、詳細ログではなく要点だけを返します。

## 小さく安全なタスクを優先する理由
この運用全体は、小さい単位を前提にすると安定します。差分が読みやすくなるだけでなく、方針変更への追従や失敗時の切り戻しが容易になり、上位幹 / 中間幹 / 枝 / Git ブランチの対応関係も維持しやすくなります。結果として、レビュー品質と実行速度を両立しやすくなります。

- 差分が読みやすく、レビュー速度と品質が上がる
- 失敗時の切り戻しが容易になる
- 方針変更があっても影響を局所化できる
- 実運用でズレが出たときも、どの階層で起きた問題かを切り分けやすい

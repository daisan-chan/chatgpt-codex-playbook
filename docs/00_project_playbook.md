# 00 プロジェクト運用プレイブック（運用ガイド）

## 目的
ChatGPT と Codex を使ったドキュメント中心のプロジェクト運用を、再現可能な手順として揃えることを目的とします。あわせて、誰が何を判断し、誰が何を実行するかを曖昧にしないことで、議論と作業の往復で運用がぶれない状態を作ります。

## 基本思想
AI を使うと生成物そのものは作りやすくなりますが、役割分担と進め方が曖昧なままだと判断の責任がぼやけます。このリポジトリでは「人が決めること」「ChatGPT が整理すること」「Codex が実行すること」を切り分け、上位幹スレッド・中間幹スレッド・枝スレッド・Git ブランチを結びつけて再現可能な運用にすることを重視します。

この 3 層モデルは、情報量の多さではなく、**どの種類の判断を扱うか**で分けます。詳細が多いから枝にするのではなく、全体方針・テーマ管理・具体タスクのどこに属する判断かで階層を決めます。

## 対象範囲
この節は、このプレイブック（運用ガイド）が日々の運用判断でどこまでを扱うかを示します。迷ったときに「まずここを見れば判断できる」範囲を明確にし、文書の役割を過不足なく保つための境界です。

- 作業の進め方
- スレッド運用とブランチ運用の対応
- 指示テンプレートとレビュー手順
- ChatGPT Project の情報源登録と Project 指示文の考え方
- 実運用でのズレの観察と改善への戻し方

## 対象外
この節は、意図的に扱わない領域を明確にして、運用ドキュメントの焦点を保つためのものです。実装や技術詳細まで広げすぎると、運用ルールの更新判断が重くなるため、ここでは共通運用に必要な情報に限定します。

- プロダクト実装の設計詳細
- 技術スタック固有のベストプラクティス集
- 自動化基盤（CI/CD）構築

## 運用モデル
運用モデルは、議論の場・実行の場・レビューの場を分離しながら連携させるための土台です。方針決定と実作業を同じ場所で混在させないことで、判断の履歴と変更の履歴をそれぞれ追いやすくします。

- **上位幹スレッド**: プロジェクト全体の方針、優先順位、完了条件、横断的な制約を扱う
- **中間幹スレッド**: 1 テーマを継続管理する。複数の枝をまたいで進捗・論点・判断保留を束ねる
- **枝スレッド**: 1 つの具体タスクを扱う。具体化・詳細化・試行錯誤・Codex 指示の受け皿にする
- **Git ブランチ / プルリク**: 枝スレッドの成果を差分として管理し、レビュー可能にする

## 3 層モデルの定義
### 上位幹の定義
上位幹は、プロジェクト全体を前に進めるための判断を集約する場です。何を先にやるか、どこまでやるか、どのテーマを継続管理するかを決めます。

### 中間幹の定義
中間幹は、1 テーマについて継続的に管理が必要になったときに作る場です。単発タスクではなく、複数の枝で扱う論点や、一定期間追跡したいテーマを束ねます。

### 枝の定義
枝は、1 つの具体タスクを進める場です。文書の具体修正、比較検討、試行錯誤、Codex への実行依頼など、局所的な作業を閉じ込めます。

## 責務と非責務
### 上位幹の責務
- 全体方針と優先順位を決める
- 中間幹を作るかどうかを決める
- 重要な制約や完了条件を保持する
- 中間幹や枝から戻ってきた要点を基に次の判断を行う

### 上位幹の非責務
- 個別タスクの詳細設計を続けること
- 試行錯誤のログを抱え込むこと
- その場で Codex へ渡す粒度の実行指示を詰めること

### 中間幹の責務
- 1 テーマの目的、論点、保留事項、関連枝の状態を継続管理する
- どの作業を枝へ切るか、どの結果を上位幹へ戻すかを整理する
- テーマ内での命名やスレッド分割が実態に合っているかを見直す

### 中間幹の非責務
- プロジェクト全体の優先順位を単独で確定すること
- 具体タスクの細部を抱え込むこと
- 枝で閉じるべき試行錯誤を引き取り続けること

### 枝の責務
- 1 つの具体タスクを完了条件に沿って進める
- 変更対象、確認方法、未解決事項を具体化する
- 結果を中間幹または上位幹へ戻せる形に要約する

### 枝の非責務
- プロジェクト全体の優先順位を決めること
- 複数テーマの継続管理を引き受けること
- 親スレッドの前提を勝手に書き換えること

## 中間幹を作る原則
中間幹は常設ではありません。**1 テーマを継続管理する必要が生じたら**作ります。具体的には、次の条件が複数当てはまるときに検討します。

- 同じテーマで複数の枝を継続的に扱う見込みがある
- 進捗・保留・論点を 1 か所で追いたい
- 上位幹に詳細を残すと全体判断が読みにくくなる
- テーマ単位で戻し先を固定したほうが運用しやすい

## 枝へ逃がす原則
枝は、**具体化・詳細化・試行錯誤が必要になったら**切り出します。上位幹や中間幹に詳細作業が流れ込み始めたら、そのまま続けず枝へ逃がすことを優先します。

- 文面修正、比較案作成、作業手順の詰め、レビュー差分対応などが始まったら枝を検討する
- 親スレッドでは「何を判断したいか」を残し、詳細作業は枝に寄せる
- 枝で得た結論は要約して親へ戻し、詳細ログは枝に残す

## 各スレッドの自己認識
各スレッドは、自身が何を扱う場かを自覚できる状態で始める必要があります。最低限、次を明示できることを原則にします。

- 自身の階層（上位幹 / 中間幹 / 枝）
- 親スレッド
- テーマ
- 目的
- 扱う範囲
- 扱わない範囲
- 戻し先
- 終着点

この自己認識は、スレッド開始時だけでなく、途中で実態が変わったときにも見直します。テンプレートは `docs/06_thread_operation_templates.md` を参照してください。

## 介入原則
ChatGPT は会話を受け身で処理するだけでなく、階層逸脱を見つけたら運用を補正する介入をしてよいものとします。ただし、勝手に構造を確定するのではなく、**提案として返す**ことを原則にします。

- **誤投入補正**: 上位幹に詳細作業が来たら、枝または中間幹への移行を提案する
- **階層逸脱補正**: 枝が全体方針を抱え始めたら、親へ戻すべき論点を切り出す
- **命名ズレ補正**: スレッド名が実態とずれたら、変更提案を行う
- **戻し支援**: 枝 → 中間幹、または中間幹 → 上位幹へ戻すときに要約を支援する

## 責任分担
責任分担は、品質だけでなく意思決定の明確さを保つための前提です。誰が何に責任を持つかを先に定めることで、AI の支援を使いつつも最終判断の所在を曖昧にしません。

- **人**: 目的設定、意思決定、承認。最終判断を人が持つことで、優先順位とリスク判断の責任を明確にする。
- **ChatGPT**: 論点整理、文書ドラフト、スレッド移行提案、レビュー支援。思考整理と叩き台作成を担い、検討の速度と網羅性を高める。
- **Codex**: 指示に基づく編集、差分提示、作業報告。実行を担い、変更内容を検証可能な形で提示してレビュー可能性を担保する。

## 標準ライフサイクル
標準ライフサイクルは、開始から報告までの最小手順を揃えるための流れです。各工程で「何をそろえるか」を明確にすることで、手戻りを減らし、次の工程に渡す情報の抜け漏れを防ぎます。

1. **開始**: 上位幹で目的・制約・完了条件を定義し、判断基準をそろえる。
2. **テーマ化判断**: 1 テーマを継続管理する必要が生じたら中間幹を作る。
3. **枝スレッドで議論**: 具体化・詳細化・試行錯誤が必要な論点を枝へ切り出す。
4. **Codex 実行**: Codex が指示どおりに編集し、差分と実施内容をそろえて提示する。
5. **レビュー**: チェックリストで整合性と過不足を確認し、方針とのズレや表記揺れを解消する。
6. **プルリク**: GitHub で差分をレビューし、目的・変更点・確認結果を記録として残す。
7. **親スレッドへ報告**: 枝は中間幹または上位幹へ、中間幹は上位幹へ、結果・判断事項・次ステップを戻す。

## 保守原則
保守原則は、更新時に迷ったときの判断基準です。新しい追記や修正を行う際は、ここに照らして「運用判断に効く情報か」「将来も維持できる粒度か」を確認します。

- 小さく安全な変更を優先する
- 既存ルールとの整合を優先し、重複記述を避ける
- テンプレートは実運用で使える最小構成を維持する
- 方針変更時は README と関連ドキュメントを同時更新する
- 実運用で出たズレは `docs/07_operational_validation.md` に蓄積し、検証後に反映する

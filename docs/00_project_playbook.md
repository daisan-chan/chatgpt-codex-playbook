# 00 プロジェクト運用プレイブック（運用ガイド）

## 目的
ChatGPT と Codex を使ったドキュメント中心のプロジェクト運用を、再現可能な手順として揃えることを目的とします。あわせて、誰が何を判断し、誰が何を実行するかを曖昧にしないことで、議論と作業の往復で運用がぶれない状態を作ります。

## 基本思想
AI を使うと生成物そのものは作りやすくなりますが、役割分担と進め方が曖昧なままだと判断の責任がぼやけます。このリポジトリでは「人が決めること」「ChatGPT が整理すること」「Codex が実行すること」を切り分け、幹スレッド・枝スレッド・Gitブランチを結びつけて再現可能な運用にすることを重視します。

## 対象範囲
この節は、このプレイブック（運用ガイド）が日々の運用判断でどこまでを扱うかを示します。迷ったときに「まずここを見れば判断できる」範囲を明確にし、文書の役割を過不足なく保つための境界です。

- 作業の進め方
- スレッド運用とブランチ運用の対応
- 指示テンプレートとレビュー手順

## 対象外
この節は、意図的に扱わない領域を明確にして、運用ドキュメントの焦点を保つためのものです。実装や技術詳細まで広げすぎると、運用ルールの更新判断が重くなるため、ここでは共通運用に必要な情報に限定します。

- プロダクト実装の設計詳細
- 技術スタック固有のベストプラクティス集
- 自動化基盤（CI/CD）構築

## 運用モデル
運用モデルは、議論の場・実行の場・レビューの場を分離しながら連携させるための土台です。方針決定と実作業を同じ場所で混在させないことで、判断の履歴と変更の履歴をそれぞれ追いやすくします。

- **幹スレッド**: 方針決定、優先順位、最終合意を扱う。全体判断をここに集約することで、個別タスクの判断基準がぶれにくくなる。
- **枝スレッド**: 個別タスクの具体化と実行指示を扱う。対象を小さく切って進めることで、検証と修正を局所化しやすくする。
- **Gitブランチ / プルリク**: 変更の実体を管理し、レビュー可能にする。会話で決めた内容を差分として残し、承認・修正・履歴管理の受け皿にする。

## 責任分担
責任分担は、品質だけでなく意思決定の明確さを保つための前提です。誰が何に責任を持つかを先に定めることで、AI の支援を使いつつも最終判断の所在を曖昧にしません。

- **人**: 目的設定、意思決定、承認。最終判断を人が持つことで、優先順位とリスク判断の責任を明確にする。
- **ChatGPT**: 論点整理、文書ドラフト、レビュー支援。思考整理と叩き台作成を担い、検討の速度と網羅性を高める。
- **Codex**: 指示に基づく編集、差分提示、作業報告。実行を担い、変更内容を検証可能な形で提示してレビュー可能性を担保する。

## 標準ライフサイクル
標準ライフサイクルは、開始から報告までの最小手順を揃えるための流れです。各工程で「何をそろえるか」を明確にすることで、手戻りを減らし、次の工程に渡す情報の抜け漏れを防ぎます。

1. **開始**: 幹スレッドで目的・制約・完了条件を定義し、判断基準をそろえる。
2. **枝スレッドで議論**: 枝スレッドで対象変更を小さく分割し、作業単位と影響範囲を限定する。
3. **Codex 実行**: Codex が指示どおりに編集し、差分と実施内容をそろえて提示する。
4. **レビュー**: チェックリストで整合性と過不足を確認し、方針とのズレや表記揺れを解消する。
5. **プルリク**: GitHub で差分をレビューし、目的・変更点・確認結果を記録として残す。
6. **幹スレッドへ報告**: 幹スレッドに結果・判断事項・次ステップを戻し、次の意思決定につなげる。

## 保守原則
保守原則は、更新時に迷ったときの判断基準です。新しい追記や修正を行う際は、ここに照らして「運用判断に効く情報か」「将来も維持できる粒度か」を確認します。

- 小さく安全な変更を優先する
- 既存ルールとの整合を優先し、重複記述を避ける
- テンプレートは実運用で使える最小構成を維持する
- 方針変更時は README と関連ドキュメントを同時更新する

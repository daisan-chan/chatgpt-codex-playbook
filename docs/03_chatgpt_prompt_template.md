# 03 ChatGPT プロンプトテンプレート

## 目的
このテンプレートは、まだ曖昧な構想や要望を `docs/02_project_brief_template.md` に沿ったプロジェクト概要へ整理したり、ChatGPT Project の指示文として参照順と介入原則を固定したりするための、ChatGPT 用テンプレートです。プロジェクト開始時に前提をそろえ、上位幹で扱う判断材料を整えやすくすることを目的にしています。Codex 実行前の整理段階で使うことを想定しています。

## まず押さえること
- Project 情報源として使うときは、登録するだけでなく**参照順**を Project 指示文に書きます
- スレッド階層は情報量ではなく**判断の種類**で分けます
- ChatGPT には、枝化・中間幹化・戻し・命名変更の**提案**を期待します
- スレッド開始時は、自己認識の最小項目を確認させます
- 詳細な運用テンプレートは `docs/06_thread_operation_templates.md` を参照します

## このテンプレートを使う場面
- プロジェクト開始時に、要望がまだ抽象的で整理しきれていないとき
- ChatGPT Project の情報源登録と指示文を整えたいとき
- 上位幹 / 中間幹 / 枝のどこで扱うべき話かを明確にしたいとき
- 新規スレッド開始時に最低限の自己認識をそろえたいとき

## ChatGPT に期待すること
- 曖昧な要望を整理する
- `目的` と `背景` を分ける
- `対象範囲` と `対象外` を分ける
- 情報不足や未確定事項を明示する
- 完了条件を確認可能な表現にする
- 後続の上位幹 / 中間幹 / 枝 / Codex 実行につながる粒度にする
- 誤投入があれば、適切な階層への移行を提案する
- スレッド名が実態とずれていれば、変更提案を行う

## ChatGPT にやってほしくないこと
- 勝手に実装詳細や技術設計へ広げること
- 未確定事項を断定すること
- 対象範囲を広げすぎること
- いきなり Codex 向け実行指示まで飛ぶこと
- 人が判断すべきことを ChatGPT が確定事項として決めてしまうこと
- 階層逸脱に気づいても、そのまま詳細作業を続けてしまうこと

## Project 指示文を書く人向けガイド
Project 指示文には、最低限次を含めます。

1. **参照優先順位**: どのファイルをどの順で読むか
2. **判断モデル**: 上位幹 / 中間幹 / 枝を何で分けるか
3. **期待する介入**: 枝化、中間幹化、戻し、命名提案をしてよいこと
4. **開始時確認項目**: 階層、親、テーマ、目的、範囲、非範囲、戻し先、終着点
5. **テンプレート導線**: 必要なら `docs/06_thread_operation_templates.md` を使うこと

## ChatGPT Project 指示文テンプレート
```md
この Project では、登録された playbook を情報源として使い、次の順で参照してください。

参照順:
1. `README.md` で全体像、役割分担、この playbook の使い方を確認する
2. `docs/00_project_playbook.md` で運用原則と 3 層モデルを確認する
3. `docs/01_threading_and_branching.md` で上位幹 / 中間幹 / 枝の使い分けと戻し方を確認する
4. 依頼内容に応じて以下を参照する
   - 要件整理: `docs/02_project_brief_template.md`
   - Project 指示文 / スレッド開始: `docs/03_chatgpt_prompt_template.md`
   - Codex 依頼: `docs/04_codex_prompt_template.md`
   - レビュー: `docs/05_review_checklist.md`
   - スレッド運用テンプレート: `docs/06_thread_operation_templates.md`
   - 実運用の観察と改善: `docs/07_operational_validation.md`

運用原則:
- スレッド階層は情報量ではなく判断の種類で分ける
- 上位幹は全体方針と優先順位を扱う
- 中間幹は 1 テーマの継続管理を扱う
- 枝は 1 つの具体タスクを扱う
- 1 テーマを継続管理する必要が生じたら中間幹作成を提案する
- 具体化・詳細化・試行錯誤が必要になったら枝化を提案する

あなたに期待する介入:
- 誤って上位幹や中間幹に詳細作業が投げ込まれたら、適切な移行先を提案する
- 枝で扱うべき話と親へ戻すべき話を区別する
- スレッド名が実態とずれたら、命名変更案を提案する
- 必要に応じて `docs/06_thread_operation_templates.md` のテンプレートを使って、開始・移行・戻しを支援する

新規スレッド開始時の確認項目:
- このスレッドの階層
- 親スレッド
- テーマ
- 目的
- 扱う範囲
- 扱わない範囲
- 戻し先
- 終着点

禁止事項:
- 未確定事項を断定しない
- 勝手に実装へ飛ばない
- 階層をまたいだ詳細作業を惰性で続けない
```

## 新規スレッド開始時の最小テンプレート
```md
このスレッドを開始するにあたり、次の項目を先に確認してください。

- 階層: 上位幹 / 中間幹 / 枝 のどれか
- 親スレッド: （なければ `なし`）
- テーマ:
- 目的:
- 扱う範囲:
- 扱わない範囲:
- 戻し先:
- 終着点:

もし会話内容がこの階層に合わない場合は、枝化・中間幹化・親への戻しのいずれを提案してください。
必要なら `docs/06_thread_operation_templates.md` の該当テンプレートを使ってください。
```

## 要件整理用のコピペプロンプト
```md
以下の構想を、`docs/02_project_brief_template.md` に沿って整理してください。

前提:
- `README.md`、`docs/00_project_playbook.md`、`docs/01_threading_and_branching.md` を先に参照し、判断の種類に応じて扱うべき階層を意識してください。
- `docs/02_project_brief_template.md` と関連サンプルが ChatGPT Project の情報源に登録されている前提で整理してください。
- もし情報源がない場合は、提示した本文と項目一覧を前提に整理してください。

指示:
- 情報不足があれば、勝手に断定せず `不明点` または `仮置きの前提` として明示してください。
- 実装詳細や技術設計には広げず、プロジェクト概要の整理に集中してください。
- `目的` と `背景` を分けてください。
- `対象範囲` と `対象外` を分けてください。
- `完了条件` は確認可能な表現にしてください。
- `ChatGPT / Codex 引き継ぎメモ` まで含めてください。
- この話を上位幹 / 中間幹 / 枝のどこで扱うべきかも補足してください。
- 最後に `次に人が判断すべきこと` を示してください。

出力形式:
1. 整理した前提
2. 不明点 / 確認したい点
3. プロジェクト概要の記入案
4. 扱うべきスレッド階層の提案
5. 対象外にしたほうがよい候補
6. 次に人が判断すべきこと

構想メモ:
- （ここに箇条書きメモや要望を書く）
```

## 使い方メモ
- 最初から全項目を埋めていなくても使えます
- 箇条書きメモや雑な構想から始めても問題ありません
- 1 回で完成させず、何回か往復して精度を上げる前提で使ってください
- 出力結果はそのまま採用せず、人が最終判断して関連文書へ反映してください
- 実務の開始・移行・戻しの定型文が必要になったら `docs/06_thread_operation_templates.md` を開いてください
